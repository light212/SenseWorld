## 1. 语音识别（ASR）服务选型

### 决策：OpenAI Whisper API

### 理由
1. **准确性高**: Whisper 在中文语音识别上表现优异，准确率可达 95%+
2. **简化架构**: 使用托管 API 无需自建语音识别基础设施
3. **成本可控**: 按使用量付费，初期用户量小时成本低
4. **快速集成**: API 简单易用，支持多种音频格式

### 评估的替代方案
| 方案 | 优点 | 缺点 | 排除原因 |
|------|------|------|----------|
| 自建 Whisper | 无 API 费用、可定制 | 需要 GPU 资源、运维复杂 | 初期规模不需要，运维成本高 |
| 阿里云 ASR | 国内访问快、中文优化 | 需要额外接入、定价较高 | OpenAI 已能满足需求 |
| Azure Speech | 企业级、多语言 | 定价较高、集成复杂 | 当前规模不需要企业级功能 |

### 技术细节
- **音频格式**: WebM/Opus（浏览器 MediaRecorder 默认格式）或转换为 MP3
- **采样率**: 16kHz 足够语音识别
- **延迟预期**: 2-3 秒（含网络传输）
- **限制**: 单次最大 25MB 音频文件

## 2. 大语言模型（LLM）服务选型

### 决策：OpenAI GPT-4

### 理由
1. **智能程度高**: GPT-4 在对话理解和生成方面表现最佳
2. **流式输出**: 原生支持 streaming，适合实时对话场景
3. **统一供应商**: 与 ASR、TTS 使用同一供应商，简化集成
4. **上下文窗口大**: 128K tokens 足以支持长对话

### 评估的替代方案
| 方案 | 优点 | 缺点 | 排除原因 |
|------|------|------|----------|
| Claude 3 | 长上下文、推理能力强 | 流式 TTS 集成复杂 | 需要额外对接不同 TTS 服务 |
| 国产大模型（通义千问等） | 国内访问快、中文优化 | 智能程度略低 | 用户体验优先，选择最佳模型 |
| 自建微调模型 | 完全可控 | 成本高、需要大量数据 | 初期不需要定制化 |

### 技术细节
- **模型版本**: gpt-4-turbo（性价比最佳）
- **上下文管理**: 保留最近 20 轮对话，总计不超过 8K tokens
- **流式输出**: 使用 `stream=True` 参数
- **系统提示**: 设置对话助手角色和行为准则

## 3. 语音合成（TTS）服务选型

### 决策：OpenAI TTS API

### 理由
1. **音质自然**: OpenAI TTS 音质接近真人，用户体验好
2. **流式支持**: 支持流式输出，可实现边生成边播放
3. **统一平台**: 减少第三方服务数量，简化运维
4. **多种声音**: 提供多种声音风格可选

### 评估的替代方案
| 方案 | 优点 | 缺点 | 排除原因 |
|------|------|------|----------|
| Azure TTS | 音质好、多语言 | 需要单独接入、延迟略高 | 增加集成复杂度 |
| ElevenLabs | 音质极佳、情感丰富 | 成本高、国内访问慢 | 成本考虑 |
| 自建 VITS | 无 API 费用 | 需要 GPU、音质可能不稳定 | 运维复杂 |

### 技术细节
- **输出格式**: MP3（兼容性最好）或 Opus（更小）
- **声音选择**: `alloy` 或 `nova`（中性、自然）
- **语速**: 1.0（标准速度）
- **流式播放**: 使用 Web Audio API 或 MediaSource Extensions

## 4. 流式传输方案

### 决策：WebSocket + Chunked Transfer

### 理由
1. **实时双向**: WebSocket 支持全双工通信，适合语音对话场景
2. **低延迟**: 避免 HTTP 请求开销，降低感知延迟
3. **原生流式**: 便于实现 LLM 流式输出和 TTS 流式播放
4. **状态管理**: 连接保持期间可维护会话状态

### 技术架构
```
用户浏览器 ←→ WebSocket ←→ 后端服务器 ←→ OpenAI APIs
     ↑                          ↓
 Web Audio API            ASR/LLM/TTS 处理
```

### 消息流程
1. **语音上传**: 用户录音完成 → Audio Blob → WebSocket 发送
2. **ASR 处理**: 后端接收 → 调用 Whisper API → 返回文字
3. **LLM 对话**: 文字输入 → 调用 GPT-4 流式输出 → 逐句返回
4. **TTS 合成**: LLM 输出句子 → 调用 TTS API → 音频流返回
5. **前端播放**: 接收音频块 → Web Audio API 播放

### 断线重连
- 指数退避重连策略
- 本地缓存未发送消息
- 自动恢复会话状态

## 5. 前端录音方案

### 决策：MediaRecorder API + Web Audio API

### 理由
1. **标准 API**: 现代浏览器原生支持，无需第三方库
2. **格式支持**: 直接输出 WebM/Opus，压缩效率高
3. **实时反馈**: Web Audio API 可提供波形可视化
4. **权限管理**: 浏览器原生权限管理，安全可靠

### 浏览器兼容性
| 浏览器 | MediaRecorder | Web Audio API | 备注 |
|--------|---------------|---------------|------|
| Chrome | ✅ | ✅ | 完整支持 |
| Safari | ✅ | ✅ | iOS 14.5+ 完整支持 |
| Firefox | ✅ | ✅ | 完整支持 |
| Edge | ✅ | ✅ | 基于 Chromium |

### 技术实现
```typescript
// 录音初始化
const stream = await navigator.mediaDevices.getUserMedia({ audio: true });
const mediaRecorder = new MediaRecorder(stream, {
  mimeType: 'audio/webm;codecs=opus'
});

// 波形可视化
const audioContext = new AudioContext();
const analyser = audioContext.createAnalyser();
const source = audioContext.createMediaStreamSource(stream);
source.connect(analyser);
```

## 6. 数据库设计

### 决策：PostgreSQL + Redis

### PostgreSQL（持久化存储）
- **用户数据**: 账户信息、偏好设置
- **对话历史**: 对话记录、消息内容
- **选择理由**: 成熟稳定、JSON 支持好、扩展性强

### Redis（缓存层）
- **会话状态**: WebSocket 连接状态
- **对话上下文**: 活跃对话的上下文缓存
- **选择理由**: 高性能、支持过期策略、适合临时数据

### 数据分层
```
┌─────────────────────────────────────┐
│           Redis (热数据)             │
│  - 活跃会话上下文                      │
│  - WebSocket 连接映射                 │
│  - 用户在线状态                        │
└─────────────────────────────────────┘
                  ↓ 持久化
┌─────────────────────────────────────┐
│        PostgreSQL (冷数据)           │
│  - 用户账户                           │
│  - 对话历史                           │
│  - 消息记录                           │
└─────────────────────────────────────┘
```

## 7. 安全与隐私

### 数据传输
- 所有通信使用 HTTPS/WSS 加密
- WebSocket 连接需要 JWT 认证

### 音频处理
- 语音数据仅在内存中处理，不持久化存储
- 转写完成后立即删除音频数据
- 仅保留转写后的文字记录

### 用户数据
- 密码使用 bcrypt 哈希存储
- JWT token 设置合理过期时间
- 支持用户删除账户和数据

## 8. 性能优化策略

### 延迟优化
1. **并行处理**: ASR 完成后立即开始 LLM 处理
2. **流式传输**: LLM 边生成边发送，TTS 边合成边播放
3. **预连接**: 页面加载时预先建立 WebSocket 连接
4. **音频预加载**: TTS 音频分块预加载

### 并发处理
1. **连接池**: 数据库和 Redis 连接池
2. **异步处理**: FastAPI 异步处理请求
3. **限流**: 单用户并发请求限制

### 缓存策略
1. **对话上下文**: Redis 缓存活跃对话
2. **用户信息**: 缓存常用用户数据
3. **静态资源**: CDN 加速前端资源
